# Blueprint: Periodic Reconciliation / Resync for MeshSync

## 1. Goal and Gap

MeshSync's discovery is **pure watch** with zero self-healing. `GetDynamicInformer` in `meshsync/meshsync.go:86` constructs the shared informer factory with a hard-coded resync period of `0`:

```go
return dynamicinformer.NewFilteredDynamicSharedInformerFactory(dynamicKubeClient, 0, v1.NamespaceAll, listOptionsFunc)
```

Zero means client-go's informer never re-delivers cached objects as synthetic `UPDATE`s and never re-LISTs. If a watch connection drops and the resumed watch has a resourceVersion gap (`410 Gone`, long GC pause, informer restart racing an API-server compaction), the object add/update/delete that happened during the gap is silently lost — MeshSync's in-memory store (`h.stores`, populated per-GVR from `iclient.Informer().GetStore()` in `internal/pipeline/step.go:59-63`) and everything downstream (Server's DB) drift from cluster truth. Today the **only** self-heal path is external: a CRD watch-list change (`meshsync/handlers.go:456-458`) or an explicit `ReSyncDiscoveryEntity` broker request (`meshsync/handlers.go:203-204,256-259`, itself driven by Server's `ResyncMeshsync` — see `meshery/server/machines/kubernetes/resync_resources.go`), neither of which fires on a timer. This is Design Objective gap #6 from the MeshSync design spec (`docs/design-spec_meshsync-infrastructure-synchronization.md:146`): *"Maintain a local snapshot of the cluster which is refreshed periodically (either through repeat scans or by watching the events stream from Kubernetes)."*

## 3. Proposed Architecture

### (a) Resync-period vs. dedicated reconcile-diff loop — decision: **dedicated reconcile-diff loop, not a non-zero informer resync period**

**Rejected: non-zero `resyncPeriod` on `NewFilteredDynamicSharedInformerFactory`.**
- It operates by replaying every cached object through the *same* `UpdateFunc` handler as a synthetic update, with `oldObj == newObj` (same resourceVersion). `internal/pipeline/handlers.go:29-32`'s `oldRV < newRV` check would correctly suppress publishing for every object that DID NOT drift — so it "does not spam Server" only by accident of that existing guard, not by design. But it fails the actual goal for two structural reasons: (1) it can only "heal" adds/updates that are still present in the informer's own local cache (a client-go resync literally iterates the local `Store`, it never talks to the API server) — an object the watch missed *entirely* (both add and delete happened in the gap, or the object was deleted while disconnected) is never in the cache to begin with, so a resync-period replay physically cannot detect it; (2) it is undifferentiated and untunable per this feature's requirement — one factory-wide period couples every GVR's "freshness" together, with no way to stagger/jitter per resource kind or scope it to the currently-filtered namespace set, and no clean way to detect the missed-delete case at all (§(c) needs a fresh LIST to diff against the local key set, which resync period does not provide).
- Conclusion: resync period is the wrong primitive for anything except catching a locally-mutated-but-unpublished cache entry (not MeshSync's failure mode, since every publish attempt is synchronous with the informer event) — it is not a re-list, and provides no missed-delete detection.

**Chosen: a dedicated, timer-driven reconcile loop that performs a fresh LIST per configured resource kind and diffs the result against the informer's own `cache.Store` (the existing per-GVR in-memory snapshot), publishing only the deltas.**

This reuses infrastructure that already exists for a different trigger (`h.stores`, the per-GVR `cache.Store` populated in `internal/pipeline/step.go:59-63` and read today only by `listStoreObjects`/`handleInformerStoreRequest`, `meshsync/handlers.go:294-311,232-254`) as the **"last-published" comparison set** — no new store needs to be introduced, and the diff computation piggybacks on the *live* informer's own cache, which stays current between reconcile ticks via the ordinary watch path. This directly satisfies §(c) (missed-delete detection) because a fresh LIST's key set, compared against the Store's current key set, surfaces objects present in the Store but absent from the LIST — those are exactly the deletes the watch missed while disconnected.

### (b) Interaction with dedup / resourceVersion suppression — how reconcile publishes ONLY drift

The existing `oldRV < newRV` guard in `internal/pipeline/handlers.go:29-46` is an **informer-event-path** mechanism (`UpdateFunc` callback wiring, comparing the informer's previous cached object to its newly-delivered one). A reconcile loop does not go through that callback at all — it runs its own LIST-vs-Store diff *outside* the informer's event stream — so `UpdateFunc`'s guard is not naturally in the loop's path and must be **reimplemented equivalently** at the reconcile layer, not bypassed:

- **Same-object, same resourceVersion (in both Store and fresh LIST)** — no drift, do not publish. This is the overwhelming majority case on every tick (nothing changed on a healthy watch) and must be a cheap key+resourceVersion comparison, not a full deep-equal.
- **Same-object, different resourceVersion (LIST's RV > Store's cached RV)** — genuine drift: the watch missed an update. Publish `broker.Update` for it (reconcile is exactly the "detected a resourceVersion advance the watch didn't deliver" case the naming already implies) and update the Store to match (informer's own event delivery will separately reconcile the Store when/if the watch itself later delivers a stale event for the same object — no double-write risk since the informer's `UpdateFunc` guard already handles that via its own RV check against whatever is in the Store at that time).
- **In fresh LIST, absent from Store** — genuine drift: the watch missed an add. Publish `broker.Add`.
- **In Store, absent from fresh LIST** — this is §(c): genuine drift, the watch missed a delete. Publish `broker.Delete`.
- Nothing else is published. This is the belt-and-suspenders design point in (e): reconcile's *entire contract* is "diff against local truth, emit only what disagrees" — it must never re-publish an object whose resourceVersion in the fresh LIST matches what's already in the Store, because that is indistinguishable from "watch is healthy and nothing happened," which is the modal case every tick.

This diff-then-publish flow bypasses the file-mode `InMemoryDeduplicator*` writers entirely (they are UID-lifetime caches for the file-output path only, not reachable from or relevant to the broker path — see §2) and does not need them: the RV-diff above **is** the dedup, computed once per object per tick rather than delegated to a downstream writer wrapper. No changes to `internal/output` are required.

### (c) Missed-delete detection

Implemented as the third branch above: for each reconciled GVR, take the key set of the informer's `cache.Store` (`store.ListKeys()` — namespace/name keyed, `client-go`'s standard cache key format) before the LIST, take the key set of the fresh LIST result, and any key present in the former and absent from the latter is a missed delete. Construct a synthetic `unstructured.Unstructured` from the *last known Store copy* of that object (it still has full `.metadata`, no live cluster copy exists to re-fetch) and route it through the identical `publishItem`/`model.ParseList(..., broker.Delete, ...)` path a real watch-delivered `DeleteFunc` already uses (`internal/pipeline/handlers.go:48-74`), so Server sees a wire-identical Delete event — no special-casing required downstream. This is also exactly the `cache.DeletedFinalStateUnknown` shape client-go itself uses for stale deletes, which the existing `DeleteFunc` (lines 57-67) already unwraps defensively — the reconcile path can either construct that same tombstone type and hand it to the existing `GetEventHandlers().DeleteFunc`, or call `publishItem` directly with the reconstructed object and `broker.Delete`; the former reuses more existing, already-tested code and is preferred.

### (d) User-tunable interval, schema-first via the MeshSync CRD, sanely defaulted

- **Default: off (0 = disabled).** MeshSync has zero periodic behavior today; introducing an always-on timer changes runtime/API-load characteristics for every existing deployment on upgrade, which the "Back-compat & rollout" rules require avoiding by default. Ship it opt-in.
- **When enabled, a conservative floor.** A minimum enforced interval (proposed 5 minutes) prevents a misconfigured CRD from turning "periodic reconcile" into "periodic re-list storm" on a large cluster — validated both at the CRD layer (`+kubebuilder:validation:Minimum`) and defensively in MeshSync itself (never trust an upstream CR value blindly, per the "never optimize for task completion at the expense of correctness" principle — a stale/misapplied CRD or bypassed admission is exactly the failure this project's error-handling rules require guarding).
- **CRD-first, not `meshery/schemas`-first**, because (per §2) this CRD has no schemas-repo representation to begin with — "schema-first" for this repo pairing means the Operator's Kubebuilder type is the source of truth that MeshSync reads, matching how `Version`/`Size`/`Broker` already flow today.
- **Field added to both `v1alpha1.MeshSyncSpec` and `v1alpha2.MeshSyncSpec` identically**, per the hard constraint in `conversion.go:26-33` — this is not optional, it is required for the existing JSON-round-trip conversion to remain lossless.

## 5. Config / Schema / CRD Changes

- **CRD is the schema of record** for this field (see §2/§4). `spec.reconcile-interval` (`metav1.Duration`, both `v1alpha1` and `v1alpha2`), `omitempty`, default absent = disabled.
- **Versioning**: added to both existing API versions identically in the same PR — no new CRD API version is needed, and doing otherwise would violate the `conversion.go` round-trip invariant.

- **MeshSync-side config**: new `ReconcileIntervalKey` in `internal/config/config.go`, populated via `RECONCILE_INTERVAL` env (set by Operator) or `-reconcileInterval` CLI flag (dev/file-mode), resolved once at `pkg/lib/meshsync.Run` startup — no runtime CRD re-poll for interval changes; a change to `spec.reconcile-interval` takes effect on the next MeshSync pod restart (Operator's SSA + rollout), consistent with how `spec.version`/`spec.size` already work (Deployment update → rollout → new pod picks up new env). This avoids adding a *second* live-reload mechanism (the existing CRD watch/`WatchCRDs` path is scoped to the resource whitelist/blacklist, not this scalar) — flagged as a deliberate simplification in §11 rather than a gap, since resource-set changes already require the same rollout-based propagation for `watch-list` today when driven by direct CR edit (only informer-level GVR set changes via the CRD-watch path get live-applied without restart, and that's a different, existing mechanism this feature does not need to replicate).

## 6. Sequencing and Feature-Flagging

The default-off interval (§3(d)) **is** the feature flag — no separate boolean is needed. Land in this order to keep every intermediate state shippable and independently testable:

1. **MeshSync**: `internal/config` plumbing (key constant, `ParseReconcileInterval`, error codes) + unit tests. No behavior change yet (nothing reads the value).
2. **MeshSync**: `internal/pipeline` refactor extracting the shared publish-drift helper, with `RegisterInformer.publishItem` delegating to it. Zero behavior change; existing tests (`handlers_test.go`) must pass unmodified — this isolates refactor risk from new-feature risk per CLAUDE.md's "separate the refactor and the fix into distinct, clearly labeled commits."
3. **MeshSync**: `reconcileLoop` implementation + `Run()` wiring + CLI flag/`Options` plumbing, gated so it never starts when the interval is `0`/unset. Unit + integration tests (§9) land here.
4. **meshery-operator**: CRD field (both API versions) + `pkg/meshsync` env-injection + controller/resource tests. Independently deployable — a MeshSync built with step 3 but run under an *old* Operator simply never receives `RECONCILE_INTERVAL` and stays disabled (safe no-op), so steps 3 and 4 do not have to land atomically, though they should land close together to avoid the field existing on the CRD with no consumer behind it for long.
5. **Docs**: `docs/agent-instructions/architecture.md` (meshsync) gets a new subsection describing the reconcile loop, its relationship to the informer resync-period-0 decision, and its relationship to the existing `ReSync` channel/debounced-restart path (they are complementary: `ReSync` = full teardown+rebuild on config change; reconcile = lightweight diff-only, no teardown). meshery-operator's `docs/architecture.md` documents the new spec field.

## 7. Back-Compat and Rollout

- **Existing CRs**: no action required (§5). **Existing MeshSync binaries running under a new Operator**: an old MeshSync binary ignores an unrecognized `RECONCILE_INTERVAL` env var entirely (never reads it) — safe.
- **Wire contract**: unaffected. Reconcile-originated events use the identical `broker.Message{ObjectType: broker.MeshSync, EventType: ..., Object: model.KubernetesResource}` shape as watch-originated events (§3(c)) — Server cannot and need not distinguish origin.
- **Rollout recommendation**: ship disabled by default across the board (Operator's Helm chart default, any sample CRs) for at least one release; document it as opt-in in release notes; consider flipping the *chart* default to a conservative non-zero value (e.g. 30m) only after this has run in production for a cycle with no re-list-storm reports — that default flip is a separate, later decision, not part of this change.
- **Rollback**: setting `spec.reconcile-interval: "0"` (or removing it) on an already-upgraded fleet disables the loop on next rollout with no other side effects, since the loop's own state (comparison keys) lives entirely in the existing `h.stores` informer caches, not in any new persisted structure.

## 8. Risks, Failure Modes, Perf/Scale

- **Re-list storm on large clusters.** A naive "list every configured GVR every tick" is the primary named risk. Mitigations designed in: (1) default-off (§3(d)); (2) enforced floor interval (5m minimum, §3(d)); (3) intra-tick staggering across GVRs with bounded concurrency (§4); (4) fleet-wide jitter via the existing `jitter()` helper (§4) so many clusters' MeshSync instances don't synchronize; (5) reconcile scope is **exactly** the informer's existing configured GVR set filtered through `GetListOptionsFunc`'s blacklist label selector (§4) — it never lists more than what's already being watched, and it respects `-outputResources` (via `outputFiltration.ResourceSet`, already applied at the pipeline-construction/write layer) so a restricted deployment reconciles only its restricted resource set. **`-outputNamespaces` is a write-time filter, not an informer/LIST-time scope** (§2) — a genuinely namespace-scoped reconcile (listing only within the allowed namespaces at the API-server level, cheaper than a cluster-wide LIST filtered client-side) is not achievable without deeper informer-factory changes (`NewFilteredDynamicSharedInformerFactory` takes one cluster-wide-or-single-namespace argument, not a set) — flagged explicitly as an out-of-scope enhancement in §11, with the interim mitigation being that write-time filtering still means a namespace-restricted deployment publishes no more drift than it would publish live events, even though its LIST cost is unreduced.
- **Dedup/RV-suppression interaction failing silently.** If the reconcile loop's own RV-diff (§3(b)) has a bug and always treats "unchanged" as "changed," it defeats the entire purpose and re-floods Server every tick. Test plan (§9) includes an explicit "steady state produces zero publishes" assertion, not just a "drift produces N publishes" assertion — the null case is equally load-bearing.
- **Missed-delete false positives from partial/paginated or rate-limited LIST failures.** A LIST that fails midway or is truncated must never be diffed as "everything not returned is deleted" — a failed/partial LIST must abort that GVR's reconcile pass for that tick (log + error code, per §4) rather than falling through to the diff, or every transient API-server hiccup becomes a false mass-delete storm published to Server. This is the single most dangerous failure mode in the design and must be a hard early-return, tested explicitly (§9).
- **Stale reconstructed object on missed-delete.** The synthesized Delete event (§3(c)) carries the last-known Store copy, not a live cluster read (the object no longer exists to read) — downstream consumers relying on delete-event payload freshness (there are none identified in Server's `Delete` handling, which only uses it for a keyed row delete, `models/meshsync_events.go:242-246`) are unaffected, but this should be called out in the PR description as an intentional, unavoidable characteristic.
- **Interaction with `debouncedRestartDiscovery`.** If a CRD-driven `ReSync` (full rebuild) fires concurrently with a reconcile tick, the reconcile loop could be diffing against a `h.stores` map that `startDiscovery` (`discovery.go:48`) is about to replace. The reconcile loop must read `h.stores` defensively (it's currently an unguarded map write from `startDiscovery` and would need a mutex or an atomic swap if reconcile runs concurrently with a rebuild) — this is a **pre-existing latent race** (`h.stores` has no synchronization today because nothing else reads it concurrently with `startDiscovery`'s write) that this feature's *first* concurrent reader exposes; per the maintainer-mindset directive to own bugs adjacent to the task, this must be fixed as part of this feature (add a `sync.RWMutex` guarding `h.stores` reads/writes) rather than left as a new landmine, even though the race predates this change.

## 9. Test Plan

**Unit (MeshSync, `go test --short ./...`)**
- `internal/config`: `ParseReconcileInterval` — zero/empty → disabled; below-floor → error; at/above floor → passthrough; malformed duration string → error.
- `internal/pipeline`: extracted publish-drift helper — table test covering Add/Update/Delete event construction identical to today's `publishItem` output (regression-guard the refactor with the existing `handlers_test.go` `recordingWriter` pattern, asserting byte-identical `model.KubernetesResource` output pre/post-refactor for the same input).
- `meshsync` (handlers): new `reconcileLoop`-focused tests using the same `dynamicfake`/`newSeededFactory`-style harness already established in `internal/pipeline/step_test.go`:
  - **Steady state**: Store and fresh LIST identical → zero writes to a `recordingWriter`.
  - **Missed add**: object in fresh LIST, absent from Store → exactly one `broker.Add`.
  - **Missed update**: object in both, LIST's resourceVersion higher → exactly one `broker.Update`, with the diffed object's fields, not the stale Store copy.
  - **Missed delete**: object in Store, absent from fresh LIST → exactly one `broker.Delete`, payload sourced from the last Store copy.
  - **Partial-LIST-failure abort**: fake client returns an error mid-GVR → zero deletes published for that GVR on that tick (guards the false-mass-delete risk in §8), plus the expected error surfaced/logged.
  - **`h.stores` concurrency**: race-detector test (`go test -race`, already the project's standard per `make test`) with a reconcile tick running concurrently with a `startDiscovery` rebuild — validates the mutex fix in §8.
  - **Disabled-by-default**: `Run()` with interval `0` never starts the goroutine (assert via a counting fake ticker or a "no LIST calls made" assertion against the fake dynamic client after a wait longer than any reasonable tick).
- `meshery-operator`: `pkg/meshsync/meshsync_test.go` — `GetServerObject` env-injection table cases for `reconcile-interval` (present/absent/zero); `api/v1alpha1/conversion_test.go` — round-trip test for the new field (mirroring the existing `TestBrokerConversionRoundTrip` shape) confirming a non-zero `ReconcileInterval` survives v1alpha1→v1alpha2→v1alpha1.

**Integration (`make integration-tests`, MeshSync)**
- Extend the existing kind+NATS docker-compose integration harness with a scenario that (1) starts MeshSync with a short (integration-test-only, sub-floor override via a test-only env or direct function call bypassing the floor) reconcile interval, (2) creates/deletes a resource while MeshSync's watch is deliberately disconnected or its informer paused (simulate via network partition to the API server, or by stopping/restarting the MeshSync process against a cluster that changed while it was down — the latter is simpler and matches the CRD-watch integration tests' existing style), (3) asserts the resulting NATS/file output contains exactly the drifted events once the reconcile tick fires, with no duplicate re-publish of untouched resources.

**Runtime verification (`verifier-meshsync` skill, per the brief's explicit requirement)**
- Follow the skill's existing pattern (`up` → `run` with `--output file` and `DEBUG=true`, drive the cluster, grep signal lines, `down`).
- **Simulated watch-gap procedure**: since MeshSync has no built-in "pause watch" hook, the practical drive is: `up` + `run`, let initial discovery complete, then `kill` (not graceful stop) the MeshSync process, mutate the cluster (create N resources, delete M pre-existing ones, patch K), restart the binary with a short `-reconcileInterval` and wait past one tick, and assert the snapshot file reflects the mutations without a second full-discovery re-list being the only mechanism that caught them (distinguish "caught by initial discovery on restart" from "caught by the periodic reconcile tick" by mutating the cluster again *after* the process is confirmed running and past its initial sync, then waiting for the tick).
- Add new signal-line entries to the skill's table (`SKILL.md`'s "Observe: signal lines" section) once the implementation lands: a debug log line marking "reconcile tick: N drifted (A adds, U updates, D deletes) out of M listed" is essential both for this verification and for production observability — the reconcile loop must log a per-tick summary even when drift count is zero, so "it ran and found nothing" is distinguishable from "it never ran."

## 10. Effort and Phasing

- **Phase 1 (MeshSync core, ~3-4 days)**: `internal/config` plumbing + `internal/pipeline` publish-drift extraction refactor + unit tests. Lowest risk, unlocks parallel work.
- **Phase 2 (MeshSync reconcile loop, ~4-6 days)**: `reconcileLoop` implementation, `h.stores` mutex fix, staggering/jitter, error handling for partial-LIST failures, unit + integration tests. Highest-complexity phase; the missed-delete diff and partial-failure-abort logic deserve the most test scrutiny.
- **Phase 3 (meshery-operator, ~2-3 days)**: CRD field (both versions) + regenerated manifests + `pkg/meshsync` env injection + tests. Can start as soon as the field name/shape is settled (end of Phase 1), independent of Phase 2's internals.
- **Phase 4 (docs + runtime verification + rollout, ~1-2 days)**: architecture-doc updates in both repos, `verifier-meshsync` skill signal-line additions, staged Helm-chart default decision (§7).
- **Total**: roughly 2-3 weeks of focused work across the two repos, phases 1-2 sequential within MeshSync, phase 3 parallelizable against phase 2 once the field contract is fixed.

## 11. Open Questions

1. **Namespace-scoped LIST cost reduction.** Should a follow-up give the reconcile loop true server-side namespace scoping (multiple per-namespace `dynamicinformer` factories, or per-namespace LIST calls) rather than relying on client-side write-time filtering, for deployments using `-outputNamespaces`? This is flagged in §8 as the main unresolved scale lever beyond staggering/jitter, and would be a larger structural change to how the informer factory is constructed (`meshsync/meshsync.go:85-87` currently takes one `v1.NamespaceAll`-or-single-namespace argument, not a set).
2. **Live-reload of the interval without a pod restart.** §5 deliberately chose restart-on-change (matching `Version`/`Size` today) over folding the interval into the existing CRD-watch live-reload path (which is scoped to the resource whitelist/blacklist only). Is that acceptable, or does product/ops expect a `spec.reconcile-interval` edit to take effect without a rollout, which would require extending `updatePipelineConfig`/`WatchCRDs` (`meshsync/handlers.go:474-528`) to also watch the `MeshSync` CR itself (not just CRDs) for this field?
3. **Should Server surface/set `spec.reconcile-interval`** through any admin UI/API, or is `kubectl`/Helm-values configuration sufficient, consistent with how `Version`/`Size`/`Broker` are configured today (§4)?
4. **Metrics.** The Operator already has a `metrics` package (`ReconcileTotal`/`ReconcileDuration`/`ReconcileErrors`, used by `MeshSyncReconciler.Reconcile`, `controllers/meshsync_controller.go:75-80`) — should MeshSync itself expose analogous Prometheus counters for reconcile-loop ticks/drift-counts/errors, and if so, where (MeshSync currently has no metrics endpoint at all, only `/healthz`/`/readyz` via `pkg/lib/meshsync/health.go`)? This is a natural observability complement but is out of this blueprint's core scope.
5. **JetStream durability sequencing.** The brief notes reconcile is "belt-and-suspenders" alongside a separate future JetStream-durability feature (broker-side redelivery/at-least-once guarantees). No JetStream work exists in this repo today (confirmed, §2). Should this feature's design leave any explicit hook (e.g. a reconcile-triggered event carrying a distinguishing marker) for a future JetStream feature to suppress redundant reconcile-vs-redelivery overlap, or is "both independently converge on correctness, occasional overlap is harmless (Server is idempotent, §2)" sufficient for now? This blueprint assumes the latter and recommends no coupling be built prematurely.
